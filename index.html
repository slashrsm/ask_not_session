Free software

Give me pros and cons

Pros: Can influence, less €€ than own, community - learning and collaboration, learning by studying code, lots of experts on the market, security, code quality, balanced efforts Cons: Costs €€, ever. avoids pushing big initiatives as someone else will do it, slow decision process

Proprietary software

Give me pros and cons

Pros: Someone does development for you, usually good docs, (pro) support, some kind of communities form Cons: No control, hard to learn if docs are not perfect, lots of €€, code quality questionable (you don't know), security, competitiveness, features.

Own* software

Give me pros and cons

*developed in house

Pros: You control, you have knowledge about it, security through obscurity Cons: All development on you (unsustainable), lost of €€, core team can leave (no knowlegde), quality of code questionable (QA, reviews, ..) - unless you have huge team, security

“We are what we repeatedly do. Excellence, therefore, is not an act, but a habit.”

Aristotle

21 days, 6 weeks, 90 days whatever. Point is that you need to repeat something for some time in order to make it a habbit.

Treat as if it would be own project

- Experience a bug: - patch -> provide feedback (at least learn how to use issue statuses - "Needs work" doesn't mean that someone will do it for free) - Check if it is reported - Report it if not (don't let bugs pass by) - **If you are doing this -> 80% important** - high prio - fix it and submit a patch - low prio - be prepared never fixed - low for you low for everyone? - habbit for your developers - **not the case - not part of the community**

Community tools are your tools

- Issue queues - IRC - Missing functionality in core/existing module - Don’t complain!!! - Feature request - Low prio: Maybe someone else does it - but be prepared that it probably won’t - High prio: Write a patch/module - Talk about it - maybe you find someone that needs it too and you share the load, collaborate (session later - Floris and Ruben) - Don’t do custom modules (unless it is very custom glue code)

Don't wait "Until it is ready"

- This is partly killing our advancement - everyone is waiting for someone else - **"Is media ready?"** - It won't happen automagically! - Requires real people to do real work!

Be selfish

- Make your solutions general so you (or someone else) can use it in the future - Share them under free sw license - Since other people will be able to use it you might get new functionality for free.

Bigger initiatives...

- are a bit different - More complex, consumes much more time - You always need to collaborate and communicate - At this point it is not just about development (blog posts, conference talks, management, motivation) - Otherwise you likely end up with project of type 5 - Start with steps above, climb the ladder and when you gain experience do this - We still don't good process for this IMO - enthusiastic individuals, burnout, not sustainable
